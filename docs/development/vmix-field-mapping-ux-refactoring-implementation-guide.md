## 1. Назначение

Этот документ — пошаговая **инструкция для программиста** по улучшению UX сопоставления полей vMix (GT) с данными приложения (dataMap), с упором на самый трудозатратный сценарий: **Rosters A/B и Lineup A/B**.

Ключевой результат: пользователь должен уметь **быстро находить** и **быстро настраивать** сопоставления даже при:

- большом числе инпутов,
- большом числе полей в инпуте,
- большом каталоге данных (особенно составы/стартовый состав),
- разных проектах vMix с разными именами инпутов и полей.

## 2. Базовые принципы реализации

### 2.1. Не привязываться к именам vMix-полей

В разных проектах vMix названия полей/инпутов отличаются. Поэтому любые ускорители настройки должны:

- работать **в рамках текущего инпута**,
- опираться на фактически загруженные поля (`getVMixInputFields(...)`),
- давать **предпросмотр** предлагаемых изменений,
- позволять применить выборочно / откатить.

### 2.2. Сначала инструменты ускорения, потом “умная автоматика”

Сначала внедряем:

- поиск/фильтры,
- группировку,
- нормальный пикер,
- копирование/вставку.

И только затем — подсказки/автозаполнение, которое может ошибаться.

### 2.3. Предпочтение TDD для критических утилит

UI можно проверять интеграционными/компонентными тестами, а “сердце” (алгоритмы подбора, mass-apply) — юнит-тестами.

### Этап 5. Специализированный ускоритель для Rosters/Lineup (приоритет №1)

**Цель:** резко сократить трудозатраты именно на 4 набора: Roster A/B и Lineup A/B.

**Статус:** ✅ MVP реализован (2026-03-26)

Важное ограничение:

- названия vMix-полей могут быть любыми, поэтому нельзя “хардкодить” список имён.

#### 5.1. Требования (что обязательно должно соблюдаться)

- **Работа только по текущему инпуту:** использовать только список фактических полей, полученный для текущего `inputId` через `getVMixInputFields(...)`.
- **Никаких жёстких ожиданий по именам:** не требовать, чтобы поля назывались `Player1Name`, `TeamA_1_Name` и т.п.
- **Предпросмотр:** до применения пользователь видит список изменений “поле vMix → dataMapKey”.
- **Идемпотентность/контроль перезаписи:** пользователь выбирает режим:
  - **только пустые** (не трогать уже сопоставленные),
  - **перезаписать** (заменить существующие сопоставления).
- **Выборочная отмена:** пользователь может снять галочки с отдельных строк предпросмотра перед применением.

#### 5.2. UX-идея (почему это ускоряет именно Rosters/Lineup)

Главная боль Rosters/Lineup — не столько “выбрать dataMapKey”, сколько сделать это **много раз**.

Поэтому ускоритель должен переводить работу из режима:

- “на каждом поле открыть блок → найти пункт в каталоге → выбрать”

в режим:

- “один раз описать структуру (A/B, roster/lineup, диапазон игроков, какие типы нужны) → быстро назначить фактические поля vMix на “слоты” → применить”.

#### 5.3. UI: модалка «Массовое сопоставление Rosters/Lineup»

**Точка входа:** кнопка в `VMixInputFieldsPanel` рядом с заголовком “Поля инпута”:

- `Массовое сопоставление…`

**Модалка состоит из 3 шагов (внутри одной формы):**

**Шаг A. Выбор источника данных (dataMap)**

- Переключатель “Набор”:
  - `Roster A`
  - `Roster B`
  - `Lineup A`
  - `Lineup B`
- Параметры:
  - диапазон игроков: `1…14` (для roster), `1…6` (для lineup) с возможностью изменить
  - какие колонки включать:
    - `Number`
    - `Name`
    - `Position (полное)`
    - `PositionShort (OH/MB/OPP/S/L)`

**Шаг B. Назначение “слотов” на фактические поля vMix**

Табличный вид (самый быстрый для серии):

- строки: `Player 1 … Player N`
- колонки: выбранные типы (`Number/Name/Position/PositionShort`)
- каждая ячейка — выбор **из списка фактических полей vMix текущего инпута**:
  - с поиском по `field.name`
  - с возможностью очистить (пусто)

Важно: это не выбор из dataMap-каталога. DataMapKey в каждой ячейке **детерминирован** выбранным набором (Roster/Lineup, A/B, индекс, тип).

**Шаг C. Предпросмотр и применение**

- список строк:
  - `[✓] <vMix field.name> → <dataMapKey> (человекопонятная подпись)`
- фильтр предпросмотра:
  - “только новые”
  - “включая перезаписи”
- режим применения:
  - “применить только пустые”
  - “перезаписать”
- кнопки:
  - `Отмена`
  - `Применить (X)` — где X = число отмеченных строк

#### 5.4. Модель данных (для реализации и тестов)

Рекомендуемые типы (в renderer):

- `type VMixField = { name: string; type: string }`
- `type MassMappingMode = 'onlyEmpty' | 'overwrite'`
- `type MassMappingSlot = { playerIndex: number; kind: 'Number' | 'Name' | 'Position' | 'PositionShort' }`
- `type MassMappingTarget = { side: 'A' | 'B'; scope: 'roster' | 'starting'; playerIndex: number; kind: MassMappingSlot['kind'] }`
- `type MassMappingAssignment = { vmixFieldName: string; target: MassMappingTarget }`
- `type MassMappingPreviewRow = { vmixFieldName: string; dataMapKey: string; willOverwrite: boolean; current?: unknown; next: unknown; enabled: boolean }`

**Функции, которые стоит вынести в отдельный модуль для тестирования:**

1) `targetToDataMapKey(target): string`

- `rosterA.player7Name`
- `startingB.player3PositionShort`

2) `buildMassMappingPreview({ assignments, currentFieldsMapping, mode }): MassMappingPreviewRow[]`

- вычисляет, какие записи будут созданы/перезаписаны
- проставляет `willOverwrite`
- применяет режим `onlyEmpty/overwrite`

3) `applyMassMapping({ previewRows, config, inputId }): nextConfig`

- применяет только строки `enabled === true`
- пишет в `config.inputs[inputId].fields[vmixFieldName] = { dataMapKey, vmixFieldType }`

Где `vmixFieldType` берётся из фактического `fields` (text/color/image) и фиксируется как сейчас.

#### 5.5. Где хранить состояние модалки

- локально в `VMixInputFieldsPanel` (state hooks), т.к. это чисто UI-операция;
- в конфиг сохраняем только итоговые сопоставления (как и сейчас).

# Автоматические плашки событий vMix (сетбол, матчбол, таймаут)

*Последнее обновление: 2026-03-26*

Единый документ: анализ, план реализации, ревью кода и описание проделанного рефакторинга.

---

## 1. Цель и контекст

**Цель:** при наступлении событий **сетбол**, **матчбол** или **таймаут** в vMix автоматически показывать соответствующую плашку и скрывать её по окончании события. Плашки должны быть видны **только когда в эфире отображается плашка со счётом**.

**Текущее состояние (до реализации):** настройка инпутов vMix, управление плашками кнопками, обновление данных из приложения; домен уже вычисляет сетбол/матчбол; опрос оверлеев раз в 2 с. Не хватало автоматического появления/исчезновения плашки и привязки к видимости плашки со счётом.

**Требования:** автоматика по событиям; зависимость от счёта (если счёт выключен — событийные плашки не показывать); данные плашек из приложения (dataMapKey, одно поле «Автособытие» с контекстом по типу события).

**Выбранный вариант:** отдельный инпут в отдельном оверлее (Вариант 1). Реализация в приложении: проверка состояния оверлея со счётом по опросу, show/hide для событийного инпута через OverlayInput*In/Out. Задержка до ~2 с при выключении счёта учтена; при смене события троттлинг и сброс по подписи дают быструю реакцию.

**Ключевые решения:** один инпут со счётом (галка в настройках); авто-инпуты с галочками по 6 типам (Сетбол А/Б, Матчбол А/Б, Таймаут А/Б); флаг `autoEventShowAlongside` (замена текста vs дополнительный оверлей); один тип события = один инпут; при «выключить счёт» в приложении — дополнительно скрывать все авто-инпуты; таймаут — состояние в приложении, кнопки «Таймаут А»/«Таймаут Б» (toggle); ручной показ авто-инпутов допускался, затем решено сделать кнопки авто-инпутов недоступными (см. рефакторинг).

**Важно (production):** настройки хранятся в `userData/settings.json` и переживают обновления. Поэтому для правила «один тип события = один инпут» добавлена автонормализация при загрузке настроек: если в сохранённом конфиге один и тот же `autoEventType` выбран в нескольких инпутах, дубликаты автоматически удаляются (тип остаётся за первым инпутом по `inputOrder`). Аналогично нормализуется `isScoreInput` (остаётся только один). Это предотвращает падение экспорта настроек из меню из‑за невалидных “легаси” конфигураций.

## 3. Доработки по багам (после внедрения)

- **Троттлинг и сброс:** минимальный интервал между проходами синхронизации 2,5 с; при смене таймаута или подписи событий (счёт/сетбол/матчбол) троттлинг сбрасывается — show/hide уходят сразу.
- **Кулдаун после hide:** 4 с — не отправлять повторный hide для того же ключа (анимация в vMix). Минимальная пауза 1,5 с между hide и show для одного ключа.
- **Не слать hide, если оверлей уже неактивен** — убраны лишние команды OverlayInput*Out при пустом оверлее.
- **Не затирать текст при завершении события:** в updateDynamicInputs не обновлять авто-инпуты, не входящие в toShow; в showOverlay не обновлять поля для авто-инпутов (обновляет только авто-синхронизация с eventType).
- **Логирование vMix API** в main (vmix-client) для отладки команд.

## 4. Проделанный рефакторинг

Проведён рефакторинг по результатам ревью: устранено дублирование, введены типы и константы, улучшена UX авто-инпутов.

### 4.1. Типы и константы (useVMix.ts)

- **Типы:** `VMixInputAutoEventConfig`, `VMixAutoEventConfig`, `AutoEventState` — единое описание конфига и результата расчёта состояния авто-событий.
- **Константы авто-оверлеев** вынесены в один блок в начале модуля с комментариями: `AUTO_SYNC_MIN_INTERVAL_MS`, `AUTO_HIDE_COOLDOWN_MS`, `MIN_MS_AFTER_HIDE_BEFORE_SHOW`.

### 4.2. Функция computeAutoEventState

- **Цель:** одна точка расчёта активных событий и списка «что показывать» вместо дублирования в эффекте и в updateDynamicInputs.
- **Реализация:** внутри useVMix добавлена чистая функция `computeAutoEventState(matchData, overlayOptions, config)` → `AutoEventState | null`. Возвращает `activeTypes`, `toShow`, `mainSlotEventType`, `eventSignature`.
- **Использование:** в `updateDynamicInputs` и в эффекте синхронизации авто-оверлеев расчёт заменён на вызов `computeAutoEventState`; в цикле используются `state.toShow`, `state.mainSlotEventType`, `state.activeTypes`, обновление троттлинга — по `state.eventSignature`.

### 4.3. Отключение ручного нажатия для авто-инпутов (VMixOverlayButtons)

- **Проблема:** ручной показ/скрытие авто-инпутов конфликтует с авто-синхронизацией и приводит к «глюкам» анимации.
- **Решение:** для инпутов с `autoEvent === true` кнопки в блоке «Управление плашками vMix» сделаны всегда недоступными (`disabled`); тултип: «Показ и скрытие управляются автоматически (сетбол, матчбол, таймаут)».

### 4.4. Итог по файлам

| Файл | Изменения |

| `src/renderer/components/VMixOverlayButtons.tsx` | Для авто-инпутов кнопка disabled, отдельный тултип. |

## 5. Ревью: быстродействие, безопасность, качество

**Быстродействие:** опрос оверлеев 2 с, троттлинг 2,5 с — приемлемо; getVMixInputFields с forceRefresh: false и кэшем в main. Последовательные await в цикле по авто-инпутам допустимы при малом их числе.

**Безопасность:** event.autoLabel разрешается только по известным eventType из AUTO_EVENT_LABELS — инъекции нет. inputKey при show/hide берётся из конфига vMix. Логирование vMix API при желании можно отключать в production (флаг/env).

**Качество:** единый тип конфига уменьшает расхождения при изменениях. Ошибки hide/show в .catch не логируются — в dev можно добавить console.warn. При необходимости — юнит-тесты на toShow/eventType для авто-оверлеев.

**Опционально на будущее:** вынос тела эффекта в runAutoOverlaySync; условное логирование в .catch для dev; отключение/сокращение логов vMix API в production.

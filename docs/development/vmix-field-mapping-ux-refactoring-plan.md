## UX-рефакторинг сопоставления полей vMix ↔ данные приложения

*Последнее обновление: 2026-03-26*

### Статус выполнения

- ✅ **Выполнено (MVP):** Rosters/Lineup mass-mapping внутри `VMixInputFieldsPanel` (адаптивно к именам полей vMix)
- ⏳ **В планах:** DataMapPicker/GTInputPicker, поиск/фильтры, copy/paste, обозреватель сопоставлений

### Контекст

На странице настроек vMix пользователь настраивает:

- список GT-инпутов (много инпутов),
- для каждого инпута — много полей,
- для каждого поля — сопоставление с «данными приложения» (dataMap).

Сейчас сопоставление выполняется через обычный HTML `<select>` со **всеми** вариантами из каталога (после «сплющивания» групп). При росте каталога (особенно «Составы/Стартовый состав») и при большом числе инпутов/полей это превращается в «огромный выпадающий список», где сложно:

- быстро найти нужный ключ,
- понять контекст (к какой группе данных относится опция),
- повторно использовать типовые сопоставления,
- отладить/найти, где уже используется тот или иной `dataMapKey`.

Точки входа в код:

- UI страницы: `src/renderer/pages/VMixSettingsPage.tsx`
- Панель сопоставления полей инпута: `src/renderer/components/VMixInputFieldsPanel.tsx`
- Каталог данных: `src/shared/dataMapCatalog.ts`
- Извлечение значения по ключу: `src/shared/getValueByDataMapKey.ts`

### Текущее поведение (кратко)

- Для выбранного GT-инпута `VMixInputFieldsPanel` получает список полей через `window.electronAPI.getVMixInputFields(...)`.
- Для каждого поля показывается «аккордеон»-блок.
- Внутри блока:
  - для `text` можно выбрать «произвольный текст» или «данные приложения»
  - для `color/image` — только «данные приложения»
  - «данные приложения» выбираются из `<select>` со всеми опциями, без поиска и без групп.

### Проблемы UX

- **Масштабируемость**: `<select>` не подходит для сотен опций; поиск по набору текста работает нестабильно и не даёт фильтра.
- **Потеря структуры**: группы каталога (`DataMapGroup`) теряются при `flattenCatalogOptions`.
- **Повторяемость действий**: однотипные сопоставления приходится делать десятки раз.
- **Нет “глобального поиска” по сопоставлениям**: невозможно быстро ответить на вопрос «куда уходит `teamA.name`?» или «какие поля ещё не сопоставлены?».
- **Список GT-инпутов в модалках тоже без поиска** (добавить/повторно сопоставить инпут).

### Цели рефакторинга

- Ускорить выбор сопоставления за счёт:
  - поиска (вводом),
  - группировки,
  - “Недавние/Частые/Избранные”.
- Снизить число кликов при массовой настройке.
- Дать “инструменты контроля”: быстро находить уже сделанные сопоставления и несопоставленные поля.
- Не ломать формат существующего конфига vMix (минимальные совместимые изменения).

### Ключевое требование: независимость от имён полей/инпутов vMix

В разных проектах vMix:

- набор GT-инпутов может быть другим,
- **имена инпутов** (`title`, `key`, `number`) могут отличаться,
- **имена полей внутри инпута** (`field.name`) могут отличаться,
- типовая структура Rosters/Lineup может быть разложена иначе.

Поэтому “автоматика” и ускорители настройки (шаблоны, подсказки, массовые операции) должны:

- **работать в рамках текущего инпута**, опираясь на фактически загруженные поля через `getVMixInputFields(...)`,
- **не требовать фиксированных названий** вида `Player1_Name` / `Team1.Text` (они могут не существовать),
- предоставлять **предпросмотр изменений** (что и куда будет сопоставлено) и позволять пользователю подтвердить/отредактировать,
- использовать гибкие правила “распознавания” (по типу поля, по токенам/шаблонам/маскам, по ранее сделанным сопоставлениям), а не “жёстко прошитые” списки.

Практический вывод: в первую очередь делаем инструменты, которые ускоряют ручную работу (поиск/группы/пикер/копирование), а “автозаполнение” строим как **адаптивный помощник**, который предлагает варианты для текущего инпута.

### Не-цели (на первом этапе)

- Полная переработка структуры `dataMapCatalog`.
- Автоматическое “идеальное” сопоставление по имени поля (можно добавить как помощь позже).

## Предлагаемый план изменений (по этапам)

### Этап 0. Быстрые улучшения без изменения модели данных (1–2 итерации)

1) **Поиск по полям инпута**

- Добавить в `VMixInputFieldsPanel` строку поиска по `field.name` (client-side).
- Добавить переключатели:
  - **Все / Сопоставленные / Несопоставленные**
  - опционально: фильтр по типу (text/color/image), если vMix возвращает типы.

2) **Группы внутри списка “Данные приложения”**

- Перестать “сплющивать” каталог.
- Рендерить `<optgroup label="Команды">...` и т.п.
- Показывать в подписи опции одновременно **label + key** (для опытных пользователей), например:
  - `Название команды A — teamA.name`

Это уже резко улучшит навигацию даже без сложных компонентов.

### Этап 1. Новый компонент выбора данных: `DataMapPicker`

Заменить `<select>` на переиспользуемый “пикер”:

- **Поиск** (вводом) по `label` и `key`.
- **Группировка** по `DataMapGroup.label`.
- **Fuzzy/substring matching** (простая реализация: includes + нормализация; затем можно улучшать).
- **Клавиатура**: стрелки/Enter/Escape.
- **Показывать**:
  - заголовок группы,
  - label,
  - key.

Интеграция:

- `VMixInputFieldsPanel` использует `DataMapPicker` для `text/color/image`.
- Модальные окна “Добавить инпут / Повторно сопоставить” получают аналогичный компонент `GTInputPicker` (поиск по title/key/number).

Технически:

- Добавить новые компоненты:
  - `src/renderer/components/DataMapPicker.tsx`
  - `src/renderer/components/GTInputPicker.tsx`
- Переиспользовать каталог через `getDataMapCatalog({ fieldType })`.

### Этап 2. “Недавние / Частые / Избранные”

#### 2.1 Недавние

- В `DataMapPicker` показывать блок “Недавние” (10–20 элементов) над результатами.
- Источник — UI-настройка (см. ниже про хранение).

#### 2.2 Частые

- Локальная статистика использования `dataMapKey` (по типу поля: text/color/image).
- Показать блок “Частые” (top 10).

#### 2.3 Избранные

- В результатах поиска кнопка “⭐” (закрепить).
- Вверху — блок “Избранные”.

#### Где хранить

Рекомендация: хранить в `settings.json` в UI-секции (не в vMix-конфиге), например:

- `ui.vmix.dataMapPicker.recentByType`
- `ui.vmix.dataMapPicker.favorites`
- `ui.vmix.dataMapPicker.usageCount`

Такое хранение:

- не ломает переносимость vMix-конфига,
- не мешает импорту/экспорту (можно включить/выключить отдельным флагом).

### Этап 3. Массовые операции для ускорения настройки

1) **Copy/Paste mappings**

- Кнопки в “Настройки инпута”:
  - “Скопировать сопоставления” (в буфер приложения)
  - “Вставить сопоставления” (с диалогом: только совпадающие поля / все / заменить существующие / только пустые)

2) **Автозаполнение по шаблону (подсказки)**

- Для поля `Team1.Text` / `Name_Team1.Text` показывать “Предложение сопоставления” (не авто-применять).
- Правила: простые маппинги по подстрокам/регэкспам, настраиваемые.

3) **Rosters/Lineup: специализированный mass-mapping (приоритет №1)**

Самая трудозатратная часть — сопоставление серийных полей для:

- Rosters A/B (много игроков × номер/имя/амплуа),
- Lineup A/B (старт + либеро).

Ключевой принцип: разные проекты vMix имеют разные `field.name`, поэтому mass-mapping должен работать как **адаптивная модалка**, которая:

- использует **фактический список полей текущего инпута**,
- позволяет пользователю назначить эти поля на “слоты” (Player 1 Name, Player 1 Number, …),
- показывает предпросмотр “поле vMix → dataMapKey”,
- умеет применять только пустые/или перезаписывать.

Подробная этапность реализации — в инструкции: `vmix-field-mapping-ux-refactoring-implementation-guide.md`.

#### Реализовано (2026-03-26)

- **UI:** кнопка `Массовое сопоставление…` + модалка выбора набора (Roster/Lineup A/B), диапазона, колонок, поиск по полям vMix.
- **Предпросмотр:** список “поле vMix → dataMapKey” с включаемыми чекбоксами (можно исключать строки).
- **Режим применения:** `onlyEmpty` / `overwrite`.
- **Дубликаты:** предупреждение, если одно и то же **vMix поле** выбрано в нескольких слотах (это приведёт к одному итоговому сопоставлению на поле).
- **Важно:** один и тот же **dataMapKey** разрешено сопоставлять на несколько разных vMix-полей (это нормальный кейс).
- **Код:** `src/renderer/components/VMixInputFieldsPanel.tsx`, `src/renderer/utils/vmixMassMapping.ts`
- **Тесты:** `tests/unit/renderer/vmixMassMapping.test.ts`, `tests/unit/renderer/VMixInputFieldsPanel.test.tsx`

### Этап 4. “Обозреватель сопоставлений” (быстро находить уже сделанное)

Добавить режим/экран:

- таблица: **Инпут (displayName/vMix title) → Поле → Тип → dataMapKey/customValue**
- глобальный поиск
- фильтры “несопоставленные”
- кнопка “Перейти к инпуту” (фокус/скролл в списке)

Это отвечает на главный вопрос: **“как более быстро находить сопоставления?”** — вместо ручного кликанья по инпутам.

## Предложение по минимальному “первому шагу” (рекомендуется)

Если нужно быстрое улучшение с минимальным риском:

- Этап 0 (поиск по полям + optgroup) + Этап 1 (DataMapPicker с поиском) без “избранных/частых”.

Это даст:

- кратное ускорение выбора,
- сохранение всей текущей логики хранения сопоставлений (`config.inputs[inputId].fields[fieldName]`),
- минимум влияния на main/useVMix.

## Тест-план (кратко)

- **UI**:
  - поиск по каталогу возвращает корректные варианты,
  - выбор корректно сохраняет `dataMapKey` / `customValue`,
  - фильтр “несопоставленные” показывает только пустые,
  - copy/paste не ломает существующие сопоставления (если внедряем).
- **Регрессия**:
  - сохранённые конфиги vMix продолжают корректно применяться в `useVMix`,
  - загрузка полей не ломается при отключённом vMix (readOnly режим).
